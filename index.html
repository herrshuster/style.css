The Semantic CSS Framework

99% Classless, 100% Semantic

The Semantic CSS Framework is a response to current web design trends, namely:

  • Semantics
  • Flat Design
  • A Focus on Typography
GitHub Link

The Problem

With the advent of CSS frameworks as a standard contention in the web industry, there has been an explosion of classism. That is to say, markup on the web is littered with classes like "col-12-3," "tiny button," and others. These classes are usually hooks into the applied CSS framework, like Foundation, Bootstrap, or Gumby. This makes it very easy for web developers to quickly prototype designs without having to go through and build each element.

However, this ruins the semantics of the web (Read More Here)The Semantic CSS Framework tries to put that right. The entire codebase uses semantic selectors to style as many elements as conceivable. Its philosophy maintains that if a pattern appears unique to a user, there is a way to define that pattern in CSS with as few classes as possible.

No More Classism!

Design Patterns

The answer to this problem is design patterns. Users will never see "button class="tiny round"," or "figure class="carousel"," but they will see a button at the end of a contact form, and they will see an image carousel. The markup for each distinct visual element will usually follow a unique pattern. For instance, a "figure > ol > li x 10 > img" is inherently a slideshow. There is no other conceivable visual element that will be a series of ordered images inside a figure. Therefore, why use classes? They can be helpful for rapid prototyping, but serve no purpose in deployment.

The Semantic CSS Framework is based on these design patterns. Its philosophy is that 99% of all conceivable elements can be selected in CSS using these design patterns. There are a number of classes, but they are semantic. By this I mean that any class names not only describe the visual appearance of the element, but also its function & content. For instance, "row" is a fairly semantic class name IF it is only used to describe a group of elements whose width is 100%. "section.colophon" is also semantic, as there is no colophon element, but colophons are to be displayed differently than other footer text. "article > aside.author" behaves in a similar fashion.

Musical Typography

One of the aims of modern web design is for more meaningful typography. However, designing with modular scales has its limitations.

First, it's predictable. Using the golden ratio to set typographic ratios is a time-tested tradition, but it results in web pages with a very familiar feel. While this may be optimal at times, what are we to do when we want to set a page's type with a different mood? Fortunately, there is already a set of ratios that is used to established moods. It's the musical scale.

The musical scale has two components: key and quality. The key of a musical scale determines the root note. Its quality defines the specific ratios that can be used to produce notes. We have similar typographic conventions. The rem unit is calculated from the "Root EleMent" font-size, and the quality can be seen as the ratios of h1-h6. The Semantic CSS framework converts established musical ratios based on these conventions.

This is not the first time this has been done. See the excellent Composing the New Canon for a more detailed exploration.

It does this through the use of SASS. When compiling the stylesheet, typographic ratios are calculated based on the ratio between notes in the chosen quality, for example domsharp9. The key sets the root element size, with C being 100%. See typography for more.
